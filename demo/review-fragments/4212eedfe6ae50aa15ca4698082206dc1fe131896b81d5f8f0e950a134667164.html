+ ## Virtual Commit 参考
120
+
121
+ 真实 commit 历史记录的是改动如何完成，却不一定是最适合评审的阅读顺序。一个很大的 AI 生成 commit 可能同时混入基础定义、高风险行为、集成、测试和生成文件。Virtual Commit 可以让 Agent 把同一份改动重新组织成依赖清晰的阅读路线，而无需 rebase、cherry-pick，也不会在仓库里创建任何 Git 对象。
122
+
123
+ 一个 Virtual Commit 是一组带名称的冻结变更块，并附有意图、1～3 条带锚点的评审重点和风险判断。它只存在于 local-mr 中：
124
+
125
+ ```text
126
+ merge base B ───────────── 选定的真实 commit H
127
+ 冻结 Diff D
128
+ │ 拆成稳定变更块
129
+ ▼
130
+ V1 → V2 → … → Vn
131
+
132
+ D 中每个 block 恰好出现一次
133
+ Vn 之后的状态等于 H 对应的冻结状态
134
+ ```
135
+
136
+ 因此，Virtual Commit 很适合审查大型或分组不理想的修改、讲清一个分支的架构，或者先展示高风险逻辑，再看机械性改动。它是阅读步骤，不是替代 Git 历史；中间虚拟状态可以不完整、不能编译，而且任何内容都不会被应用回 Git。

+ ### 工作原理
139
+
140
+ 1. `snapshot` 精确冻结从目标分支 merge base 到某个选定真实 commit 的已提交比较。工作区端点会被拒绝，因此暂存、未暂存、未跟踪和其他未提交改动无法进入 source。
141
+ 2. 普通文本修改会被拆成稳定的连续变更块；重命名、二进制、submodule、mode 变化和其他不适合安全拆分的修改，会保留成完整的文件级 block。
142
+ 3. Agent 按需读取文件/block 目录，并生成 manifest，把每个 block 分配到一个有序 Virtual Commit。
143
+ 4. local-mr 会拒绝未知、遗漏、重复 block 和无效评审锚点；随后在隔离的临时 Git 仓库中物化所有累积状态，验证最终 tree 等于冻结目标 tree。
144
+ 5. 通过校验的计划会保存为不可变 revision，并在与 Real 审查相同的 Diff 工作区打开。之后的调整会追加 revision，不会覆盖历史；每个 revision 都保留生成它时使用的精确冻结分支 commit。
145
+
146
+ Source 会记录精确的 base、head、target SHA、规范化 patch 的 SHA-256，以及受影响文件的内容寻址副本。Real 与 Virtual 使用相同的 binary/full-index diff 格式，所以它们的**完整范围拥有相同 patch 和渲染后 Diff**。任意局部范围不要求相同：Real 遵循 first-parent Git 历史，Virtual 则有意遵循另一套阅读顺序。
147
+
148
+ 整个流程只是只读投影，不会修改工作区、真实 index、对象库、refs、commit、Git 配置、远端或远端 MR。

+ ### 对照 Real 与 Virtual
171
+
172
+ 来源切换器属于每一个评审页面，因此直接运行 `local-mr` 和打开 Virtual Review 链接看到的始终是同一个工作区。裸命令对应“仓库 + 分支 + 目标分支”的唯一 live 页面，并会自动发现这个身份下已经保存的 Virtual Review。之后即使分支继续提交，旧计划也不会消失：入口会显示 **Virtual · stale**；进入 Virtual 后，常驻提示会同时标出冻结 commit 与当前 `HEAD`，并明确说明后续提交不在这份计划里。没有匹配计划时，Virtual 一侧仍会显示并明确标为不可用，而不是让整个页面换一副样子。两者共用改动树、按需加载、双栏/单栏布局、语法高亮、已读标记、上下文展开、Markdown/Mermaid 预览、移动端抽屉，以及严格相同的两个比较模式：
173
+
174
+ - **Single commit** 可跳转当前来源顺序中的任意 commit。对 `Vn` 而言，它比较累积到 `Vn-1` 后的状态与累积到 `Vn` 后的状态。
175
+ - **Commit range** 比较一段包含首尾的连续范围。对 `V3…V6` 而言，左侧是 `V2` 之后的状态，右侧是 `V6` 之后的状态。单 commit 范围同样合法。
176
+
177
+ Real 默认打开完整已提交的 **Commit range**，遵循 first-parent 顺序，并可把工作区显式作为最后一个检查点。Real range 从第一个评审 commit 开始时固定以 merge base 为左边界，避免目标分支被合入 feature 分支后，target-only 修改重新出现在审查中。Virtual 默认以 **Single commit** 从 `V1` 开始，再按 `V1`、`V2`……顺序提供前后导航、revision 选择和各 revision 独立的已评审进度。Revision 选择器会用 SHA、subject 和分支标明每份计划对应的冻结分支 commit；同一个 review 的不同 revision 可以有意指向不同 commit。共享上下文面板会显示真实 Git subject/body，或 Virtual 的意图、评审重点、风险和带锚点指引。
178
+
179
+ 如果从 live Real 页面进入一份过期计划，其中的 Real 按钮会回到同一个 live 页面；直接打开历史 Virtual 链接时，仍会使用 frozen Real 做精确对照。从 frozen Real 切回 Virtual 时，会精确保留 revision、模式、端点和当前文件。Frozen Real 使用 snapshot 的 base、head、target SHA 打开，因此仓库后续漂移或陈旧的 target 配置不会静默改变文件集合。页面不再有 Pushes 模式；旧 `mode=push` URL 只作为别名兼容，并会规范化为 Single commit 或 Commit range。
